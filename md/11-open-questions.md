# 11 · 开放问题（已确认）

> 以下决策已于 2026-07-22 确认，已同步至各规划文档。Phase 1 可据此启动。

## A. 产品定位

### A1. 应用形态 ✅

**决策：**
- **当前：** Electron 桌面应用（本地存稿、隐私优先）
- **后续：** 可能扩展小程序端
- 架构预留 IPC / 核心逻辑与 UI 分离，便于未来移植

---

### A2. 与 v2 的关系 ✅

**决策：**
- v3 **完全独立**新项目，**不提供**从 v2 导入章节正文
- v2「主角面板/物品技能」**不保留为独立模块**
- 主角面板能力**归入角色卡片**：每个角色拥有类似面板的词条区（功法/技能、法宝/装备、年龄/寿命等）
- 角色面板**不**在识别后智能自动更改——仅通过识别预览确认流程写入，或作者手动编辑

---

### A3. 首页定义 ✅（2026-07-22 修订）

**决策：方案 A 变体**
- 打开作品 → **默认进入「正文」**，不进入地图
- 地图在侧栏「地图」独立查看
- 地图可视化：**文本 LLM 生成地图代码**（HTML/SVG），沙箱 iframe 渲染
- 地图须表达**真实地理关系**（方位、邻接、远近），非无关联图片拼贴
- 地点区域可**悬停突出**；左侧：**世界选择器** + 搜索/查询
- 地图内容由作者**提示词**驱动；**共用文本 LLM**，不需另接图像 API

> ~~原方案：图像 API 生成独立板块~~ → 改为代码渲染，更好表达地理结构。

## B. 角色与识别

### B1. 主角关系远近量化 ✅

**决策：识别时必须量化，写入关系网布局依据**

| 项目 | 规定 |
|------|------|
| 输出时机 | 非主角角色，正文有明确关系描写时**必须**输出 |
| 输出内容 | `与主角关系`（类型）+ `与主角关系远近`（整数 **1–5**）+ excerpt |
| 分值含义 | 5=极近核心，4=很近，3=一般，2=较远，1=极远（详见 `06` §10.3） |
| 模糊时 | 默认 3 分，作者在预览 name\|value 表中修改 |
| 无依据时 | 不输出，禁止猜测填 4–5 分 |
| 落库 | commit 后写入 `Character.protagonistRelation`，驱动关系网节点距中心半径 |
| 群像模式 | 仍相对 `Project.protagonistId` 量化；多中心用 `isNetworkCenter` 单独标记 |

---

### B2. 识别字段范围 ✅

**决策：以下字段 MVP 必须识别（Step 1 + Step 2）**

| 阶段 | 字段 |
|------|------|
| Step 1 | 角色名、昵称/称呼/别名 |
| Step 2 | 身份/称号、境界、所在地、势力、关系（角色间）、功法/技能、法宝/装备、年龄/寿命、状态 |
| Step 2 | 与主角关系 + 与主角关系远近（见 B1） |

- 角色面板词条：匹配作者预定义模板 key 时一并提取
- **只预览，不建议：** 仅展示提取结果，不提供 AI 创作建议
- 完整规范见 `06-text-recognition-pipeline.md` §10

---

### B3. 确认流程严格度 ✅

**决策：严格模式**
- **任何**识别结果都必须经预览确认后才写入角色库
- **不提供**宽松模式（高置信度自动合并）
- 不在设置中提供模式切换

---

### B1b. 主角数量与关系网 ✅

- **视觉默认：** 单主角居中
- **群像模式：** 可多中心关系网（作者切换）
- **预览可编辑：** 识别结果在 name\|value 表中可修改后再确认

---

### B4. 同名角色 ✅

**决策：作者绝对裁决，系统禁止自动消歧**

| 项目 | 规定 |
|------|------|
| 触发条件 | 库中多个同名、上下文无法唯一匹配、置信度非 high 且候选≥2 |
| 系统行为 | 进入 `ambiguousNames`，**阻断 Step 2**，弹出裁决 UI |
| 作者操作 | 选择已有角色 / 新建（**必填** disambiguation）/ 跳过该提及 |
| 禁止行为 | AI 自动选定同名角色、静默合并、未裁决即 commit |
| 建卡 | 库中已有同名时，新建必须填 disambiguation（如「青云宗·张三」） |
| 完整规范 | 见 `06-text-recognition-pipeline.md` §10.4 |

## C. 地图

### C1. 地图可视化深度 ✅（2026-07-22 修订）

**决策：**
- **MVP 主视觉：** 文本 LLM 生成 HTML/SVG 地图代码 → 沙箱渲染，表达真实地理关系
- **MVP 辅助：** 树状层级 + 左侧世界选择 + 搜索
- **交互：** 地点区域悬停突出，树与地图双向联动
- **第二期：** 增量补丁生成（增删地点后局部更新代码）

> ~~原方案：图像 API 板块~~ → 改为代码渲染。

## E. 技术与隐私

### E1. LLM 使用方式 ✅（2026-07-22 修订）

**决策：**
- **用户自备 API Key**（OpenAI 兼容接口）
- **文本 LLM 一份 Key 两用：** 章节识别 + 地图代码生成
- MVP 不内置国内模型、不集成本地小模型（Ollama）
- ~~不需要另接图像生成 API~~

---

### E2. 发送给 API 的内容 ✅

**决策：**
- **正文识别 Step 1：** 仅发送**当前章节正文**（不向 API 发送角色库名表或快照）
- **正文识别 Step 2：** 按本章登场角色**逐人**发送该角色的 current 字段 + Step 1 对该角色的本章提取
- **地图生成：** **单独发送**地图提示词 + 节点列表，不与章节识别混发；共用文本 LLM
- 不发送全书正文；**禁止**一次发送全书角色库快照

- 默认建议路径可提示，但不强制

## I. 正文识别流程补充（2026-07-22 确认）

### I1. 两步识别 ✅

1. **Step 1：** 识别角色名 + 昵称/称呼，根据上下文匹配角色库
2. **Step 2：** 对已匹配角色提取字段信息
3. Step 1 歧义未清完不进入 Step 2

### I2. 预览形态 ✅

- 从角色库拉取已有信息 → **上文**只读展示
- 本章识别结果 → **下文** name | value 两列表格，可编辑
- 每行 checkbox，变更行高亮

### I3. 存储与更新规则 ✅

| 规则 | 说明 |
|------|------|
| 章节只存正文 | 预览**不入库**，仅内存暂存 |
| 仅最新章可 commit | 非最新章可识别预览，但禁止「确认更新」 |
| 大幅修改 | 删该章之后所有章节 → 重新粘贴/上传 → 重新识别 |
| 出场章节 | commit 时记录角色在哪些章出现 |
| 时间线 | 出场时间线 + 字段时间线（history 按章） |
